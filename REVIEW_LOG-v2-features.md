2. [ ] 모든 에러 흐름(sad path)이 명세되어 있는가?
   - **심각**: 대시보드에서 프로파일이 삭제된 경우 해당 셀의 동작 미정의. 스트리밍 중 파일 포맷이 변경되면(헤더 변경)? 컬럼 생성에서 0으로 나누기(division by zero)? 내보내기 중 디스크 공간 부족? 이 모든 에러 흐름이 누락됨
3. [ ] 빈 상태(데이터 없음, 선택 없음)에서의 동작이 정의되어 있는가?
   - 대시보드 빈 셀은 정의됨(FR-1.7). 그러나 **데이터셋이 아예 없는 상태에서 대시보드 활성화 시 동작 미정의**. 주석 패널에서 주석이 0개일 때 빈 상태 UI 미정의. 스트리밍에서 빈 파일(0행)을 감시할 때 동작 미정의
4. [ ] 취소/중단 시나리오가 처리되어 있는가?
   - **심각**: 컬럼 생성 중 취소하면? 내보내기 중(특히 대시보드 PDF) 취소 버튼이 있는가? 대시보드 레이아웃 편집 중 취소하면 이전 상태로 복귀하는가? 전부 미정의
5. [ ] 중복 호출(더블클릭, 연속 호출) 시 안전한가?
   - 테마 토글 연타 시 상태 일관성 미보장. 내보내기 버튼 더블클릭 시 중복 파일 생성 가능. "Create" 버튼(컬럼 생성) 더블클릭 시 중복 컬럼 생성 가능
6. [x] 기존 기능과의 상호작용에서 충돌이 없는가?
   - 프로파일 비교 모드와 대시보드 모드의 관계는 배타적인지 공존 가능한지 **불명확하지만**, 기존 아키텍처(ProfileController, ViewSyncManager)를 활용하는 구조이므로 부분적으로 인정
7. [ ] 되돌리기(Undo) 가능 여부가 명확한가?
   - **심각**: Cmd+Z(Undo)가 단축키에 포함되어 있지만, **어떤 동작이 Undo 가능한지 전혀 정의되지 않음**. 주석 삭제를 Undo? 컬럼 생성을 Undo? 대시보드 레이아웃 변경을 Undo? 테마 전환은? Undo 스택의 범위와 깊이가 완전히 누락
8. [ ] 경계값(0, 1, MAX, MAX+1)에서의 동작이 정의되어 있는가?
   - 대시보드 격자가 3×3 최대인데 셀 스팬으로 3×3을 초과하려 하면? 이동평균 window=0이나 window=데이터행수+1일 때? 폴링 간격 0.5초 미만/60초 초과 입력 시? 주석 텍스트 201자 입력 시? 경계값 처리 대부분 누락
9. [ ] 사용자에게 보이는 모든 에러 메시지가 명확한가?
   - FR-3.9에서 수식 오류 하이라이트만 언급. 나머지 6개 기능의 에러 메시지 형식/내용이 전혀 정의되지 않음
10. [ ] 기능 ON/OFF 전환 시 상태 정리가 되는가?
    - 스트리밍 비활성화 시 FileWatcher 정리는 암시적이나 명시적이지 않음. **대시보드 모드 종료 시 셀에 있던 차트 위젯은 어떻게 되는가?** 메모리에서 해제? 숨김? 재진입 시 복원?

### 🏗️ Structure Reviewer
**페르소나**: 15년차 시스템 아키텍트. 클린 아키텍처와 SOLID 원칙을 체화한 사람.

**Status**: ❌ **REJECT**

**체크리스트:**
1. [ ] 새 코드가 기존 아키텍처 패턴과 일관되는가?
   - 기존 아키텍처(core/ → state, engine, controller | ui/ → panels, views)는 설명되어 있으나, **새 7개 기능의 모듈이 어디에 위치하는지 명시되지 않음**. DashboardLayout은 core? ui? FileWatcher는 core? 새 패널/다이얼로그의 파일 경로?
2. [ ] 의존성 방향이 올바른가? (core ← ui, 역방향 없음)
   - **FileWatcher가 QObject를 상속하고 Signal을 사용** — 이건 core 모듈인데 Qt 의존성이 있음. 기존 패턴에서 core는 Qt-free인지 Qt 의존인지 불명확. DashboardLayout은 순수 데이터인데 어디에 둘 것인지?
3. [x] 순환 의존성이 발생하지 않는가?
   - 데이터 구조만으로는 순환 의존성 없어 보이나, 구현 단계에서 주의 필요
4. [ ] 단일 책임 원칙을 지키고 있는가?
   - **AppState가 이미 전역 상태를 관리하는데, 7개 기능의 상태가 모두 추가되면 God Object화 위험**. 대시보드 상태, 스트리밍 상태, 주석 상태, 테마 상태, 단축키 설정이 전부 어디서 관리되는지 미정의
5. [ ] 인터페이스/추상화가 적절한가?
   - **대시보드 셀에 들어가는 위젯의 인터페이스가 미정의**. 현재는 차트만이지만, Q10에서 "향후 확장"을 언급함. 그렇다면 CellWidget 같은 추상화가 필요하나 없음. ComputedColumn의 kind별 처리가 if/elif 체인으로 갈 가능성 높음 — Strategy 패턴 필요
6. [ ] 새 모듈 추가 시 기존 파일 수정이 최소화되는 구조인가?
   - **MainWindow에 7개 기능이 직접 연결될 가능성 높음**. 현재 PRD에서 기능별 컨트롤러/매니저 계층이 정의되지 않아, main_window.py가 비대해질 위험
7. [ ] Signal/Slot 연결이 명확하고 해제 시점이 정의되어 있는가?
   - FileWatcher의 Signal 3개만 정의됨. **대시보드 셀 간 동기화 Signal, 주석 변경 Signal, 테마 변경 Signal 등이 누락**. 해제 시점은 전혀 논의 안 됨
8. [ ] 파일/클래스/메서드 네이밍이 기존 컨벤션과 일치하는가?
   - 기존: `profile_store.py`, `profile_controller.py` (snake_case, 역할 명확). 새 모듈 네이밍이 정의되지 않음
9. [x] 공통 코드 중복 없이 재사용되고 있는가?
   - Phase C에서 "MiniGraphWidget 재사용" 언급으로 재사용 의도는 있음
10. [ ] 향후 기능 확장을 위한 확장 포인트가 설계되어 있는가?
    - 대시보드에 차트 외 위젯 추가(Q1), 네트워크 데이터 소스(제외 항목), 플러그인 시스템(제외)을 향후로 미뤘는데 **확장 포인트가 설계에 반영되어 있지 않음**

**상세 피드백:**

**강점:**
- 기존 아키텍처를 섹션 2에 명확히 문서화
- 구현 순서(Phase A→B→C)의 의존성 분석이 합리적
- 데이터 구조(@dataclass)가 깔끔하게 정의됨

**주요 이슈:**
1. **모듈 배치 계획 완전 누락 (Critical)**: 새 기능들이 기존 `core/`, `ui/` 구조 어디에 들어가는지 파일 단위 매핑이 없음. 이건 구현 전에 반드시 정의해야 함

3. **컨트롤러 계층 부재**: 기존에 ProfileController, ProfileComparisonController가 있는데 새 기능에 대한 컨트롤러(DashboardController, StreamingController 등)가 정의되지 않음

   - **완전 누락**: 대시보드 9셀 초기 렌더링 중, 컬럼 생성 계산 중(최대 500ms), 내보내기 중(PDF는 수 초 가능), 대용량 파일 리로드 중 — 어떤 로딩 UI도 정의되지 않음

5. [ ] 단축키가 기존 단축키와 충돌하지 않는가?

6. [ ] 창 크기 변경 시 레이아웃이 깨지지 않는가?

7. [x] 색상이 다크/라이트 테마 모두에서 가독성이 좋은가?
   - 기존 Tailwind 테마 활용 + 차트 색상 동기화(FR-6.5) 명시. 주석 색상(🔴🟡🟢🔵)의 테마별 가시성은 검토 필요하나 기본적으로 충족
8. [ ] 버튼/아이콘의 의미가 직관적인가?
   - 대시보드 ⚙와 ✕는 직관적. 그러나 **스트리밍 ▶/⏸ 토글의 초기 상태가 뭔지 불명확**. "Follow tail" 체크박스의 위치가 "차트 오른쪽 하단"이라는데 줌/팬 컨트롤과의 시각적 충돌 가능성
9. [ ] 애니메이션/전환이 부드럽고 16ms 프레임 내인가?
   - NFR-6.6에서 "깜빡임 없이 부드러운 전환"이라고만 함. **대시보드 모드 진입/퇴장 전환 애니메이션이 정의되지 않음**. 갑자기 레이아웃이 변경되면 디소리엔팅(disorienting)할 수 있음
10. [ ] 기존 UI 컴포넌트와 스타일이 일관되는가?
    - 새 다이얼로그(컬럼 생성, 내보내기 설정, 단축키 도움말)의 **크기, 버튼 위치, 패딩, 폰트 스타일이 기존 다이얼로그와 일치해야 하는데 구체적 스타일 가이드가 없음**

**상세 피드백:**

**강점:**
- ASCII 와이어프레임으로 UI 구조가 시각화됨
- 상태바 스트리밍 표시(🟢/⏸/⏹)가 정보 밀도가 좋음
- 컬럼 생성 다이얼로그에 프리뷰가 포함됨 — 좋은 UX

**주요 이슈:**
1. **로딩/프로그레스 UI 전면 부재 (Critical)**: 500ms 이상 소요되는 모든 작업에 로딩 표시가 필요. 컬럼 생성, 대시보드 렌더링, 내보내기 전부
2. **대시보드 반응형 정책 미정의**: 최소 셀 크기, 최소 창 크기, 스크롤 정책이 없음
3. **macOS 단축키 충돌**: Cmd+D, Cmd+L 등이 시스템 또는 Qt 기본 단축키와 충돌 가능
4. **접근성(Accessibility) 전면 누락**: 스크린리더 대응, 고대비 모드, 색맹 사용자 대응(주석 색상이 색으로만 구분됨)이 전혀 없음
5. **주석 UI 인터랙션 디테일 부족**: 차트 위 데이터 포인트 클릭으로 주석 추가 — 줌/팬 모드일 때 클릭과 어떻게 구분? 모드 전환(주석 모드 vs 탐색 모드)이 필요하지 않은가?

   - 📌 "Annotate" 툴바 버튼의 "주황색 강조"는 다크/라이트 양쪽에서 가시성 양호한 색상. 모드 뱃지도 동일 주황색 계열이면 일관성 유지. crosshair 커서는 OS 네이티브이므로 테마별 자동 대응
8. [x] 버튼/아이콘의 의미가 직관적인가?
   - 📌 핀 아이콘 = "표시/주석" 개념과 직관적으로 연결. "Annotate"라는 텍스트 라벨이 함께 있어 아이콘만으로 의미를 추측할 필요 없음. crosshair(✚) 커서는 "배치/선택" 모드의 업계 표준 관례 (Figma, Photoshop, Google Sheets 등). grab(🖐️) 커서는 "탐색/이동" 모드의 업계 표준 관례 — 사용자가 별도 학습 없이 현재 모드를 인지 가능
9. [x] 애니메이션/전환이 부드럽고 16ms 프레임 내인가?
   - 모드 전환 시 변경 요소: (1) 커서 변경 — OS 즉시 반영, (2) 뱃지 표시/숨김 — DOM/위젯 show/hide로 1프레임 내, (3) 툴바 버튼 강조 — CSS/스타일시트 토글로 1프레임 내. 모든 전환이 렌더링 파이프라인 내에서 완료 가능. Round 2에서 지적한 "대시보드 모드 전환 애니메이션"과 달리, 주석 모드 전환은 레이아웃 변경이 아닌 인터랙션 모드 변경이므로 fade/slide 없이 즉시 전환이 적절
10. [x] 기존 UI 컴포넌트와 스타일이 일관되는가?
    - 툴바 토글 버튼 패턴이 스트리밍 ▶/⏸ 토글과 동일한 활성/비활성 토글 패턴. 차트 내 뱃지는 새 요소이나, Section 5.4 와이어프레임에서 차트 우측 상단에 명확히 정의되어 있음. 주석 추가 다이얼로그는 기존 다이얼로그 패턴(텍스트 입력 + Cancel/OK 버튼)과 일치

**집중 검토 항목 상세 평가:**

1. **주석 모드 ↔ 탐색 모드 전환 UX가 직관적인가?**
   - ✅ **직관적**. 세 가지 진입 경로가 사용자 숙련도별로 계층화됨:
     - **초보자**: 툴바 📌 버튼 — 눈에 보이고 클릭만 하면 됨
     - **중급자**: `Cmd+Shift+A` — 키보드 중심 워크플로우
     - **고급자**: `Shift+클릭` — 모드 전환 없이 즉시 주석 (최소 마찰)
   - 퇴장도 직관적: 주석 완료 시 자동 복귀, 또는 동일 버튼/단축키로 수동 퇴장

2. **커서 변경/뱃지로 현재 모드를 충분히 인지할 수 있는가?**
   - ✅ **충분**. 이중 피드백(즉각적 + 지속적)이 핵심:
     - **커서 변경**: 즉각적이고 마우스 위치에 항상 동반 — "지금 뭘 하는 중인지" 즉시 인지
     - **뱃지**: 지속적이고 차트 코너에 고정 — 커서가 차트 밖에 있어도 모드 확인 가능
   - grab↔crosshair 구분은 업계 표준(Figma hand tool ↔ frame tool, Photoshop move ↔ marquee)이므로 학습 비용 최소

3. **Shift+클릭 일시 주석, Shift+OK 연속 주석이 사용자에게 discoverable한가?**
   - ⚠️ **부분적**. Shift+클릭과 Shift+OK는 power-user 패턴으로, 자체적으로 discoverable하지 않음. 그러나:
     - **주요 경로(툴바 버튼, 단축키)가 충분히 discoverable**하므로 핵심 워크플로우에 지장 없음
     - 툴바 📌 버튼의 **툴팁에 "Shift+Click for quick annotation"** 힌트를 포함하면 discoverability 향상 가능
     - 주석 다이얼로그 OK 버튼 옆에 **"Hold Shift to continue annotating"** 힌트 텍스트로 보완 가능
     - 이는 구현 시 쉽게 추가할 수 있는 Minor 개선이며, 기본 설계를 REJECT할 수준은 아님

4. **대시보드 모드에서 셀 변경 시 자동 해제가 혼란스럽지 않은가?**
   - ✅ **혼란스럽지 않음**. 이유:
     - 셀 변경은 **컨텍스트 전환**이므로 도구 모드 초기화가 자연스러운 기대 동작 (Photoshop에서 레이어 전환 시 선택 해제와 유사)
     - 뱃지가 사라지고 커서가 grab으로 복귀하므로 **모드 해제가 시각적으로 명확**
     - 새 셀에서 주석을 추가하려면 다시 📌을 누르면 됨 — 1클릭 비용이므로 마찰이 낮음
     - 만약 자동 해제하지 않으면, 사용자가 다른 셀에서 의도치 않게 주석을 추가하는 사고가 더 위험

- 모드별 클릭 동작 정의에서 "±5px snap" 등 구체적 수치가 포함되어 구현 모호성 최소화
